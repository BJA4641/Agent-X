## 0 · THE HEADLINE FINDING

> **The platform has never published a single piece of content — and the reason is now identified with certainty.
> It is not the pipeline, not the agents, not the budget. It is a missing escalation rung.**

Live evidence, `jobs` table, 8 queued `creative.write_script` jobs, identical error:

```
no model: writer: no script — council: all free drafts failed ->
  gemini/gemini-2.5-flash: HTTP Error 429: Too Many Requests |
  groq/llama-3.3-70b-versatile: HTTP Error 403: Forbidden
```

The writer's fallback ladder is: *free provider A → free provider B → **delay 30 minutes and try again***
(`creative.py:244-248`). There is **no rung that escalates to a paid model when free tiers are exhausted**,
even though: the kill switch is OFF, cost mode is `normal`, today's spend is **$1.15 against a $25/account/month
cap**, and paid keys are present in `provider_inventory`.

Consequence: Gemini rate-limits (429) and Groq rejects the key (403), so every script defers half an hour,
forever. `board_items` = **7,408 cleared · 4 idea · 6 rejected · 3 failed · 0 approved · 0 scheduled · 0 published (all time)**.
184 board rows carry a `script` payload historically, proving the pipeline *can* write — it simply cannot
finish while its only two writing hands are rate-limited and its wallet is untouched.

**This single defect explains ~90% of the business failure to date.** It is the entire content of the
recommended Batch #2. Groq's 403 is a second, independent blocker (REQ-FREEKEY-1, founder-actionable now).

**Operating model.** A solo non-technical founder directing an AI engineering function, deploying via web UI

committee. That is a genuine structural advantage and it shows in shipping velocity — five releases (5.8.5 → 5.9.5)
inside roughly one week, each addressing a *measured* defect rather than a guessed one.

**What is working organizationally**

- **Institutional memory now survives context loss.** `docs/LEDGER.md` in-repo (DEC-026) was the correct response to losing three hours of audit work to a truncated chat. That is an organization learning from its own failure mode.
- **Ethical guardrails are enforced in code, not policy documents.** No-income-claims, $0 mandate for paused accounts backed by a *failing test*, honest agent behaviour after the v1.5 fake-QA incident.

**What is at risk organizationally**
- **Single point of failure: you.** Every deploy, key, and approval routes through one person. A week of unavailability halts the company.
- **Verification debt.** The founder cannot independently verify agent claims, which makes the honesty of the reporting layer load-bearing. N-3 (health row lying) is exactly the class of failure that erodes it.
- **Unresolved brand risk.** The "Agent-X" / agentx.so trademark collision remains open and blocks public marketing (REQ-TRADEMARK 🔵).
- **Success has no operational plan.** If the escalation fix lands and 105 accounts start producing, the current single worker cannot serve them, and approvals still funnel through one human desk.

**Organizational grade: 6.5/10** — exceptional decision speed and engineering discipline for a solo operation;
concentrated key-person risk and no verified path from "it produces" to "it operates at scale."

## 6 · ENGINEERING ASSESSMENT

**Strengths (genuinely above bar)**
1. **Queue design.** Postgres-backed, CAS claim, idempotency keys, retry caps, dead-letter, priority + fairness ordering. 99.93% job success across 39,150 jobs.
2. **Failure-derived testing.** Tests encode named production incidents (identity-column rejection, handler arity, empty topic, $0 guarantee). This is how mature teams test.
3. **Cost governance in depth.** Four independent brakes — kill switch, daily budget, $25/account monthly cap, degrade-to-free — layered rather than duplicated.
4. **Honest instrumentation.** `run_ledger` attributes real dollars per item; the event bus reports refusals and no-ops, not just successes.
5. **Recovery engineering.** Boot orphan reaping, stuck-job self-heal, soft pause, circuit breakers, windowed idempotency to make floods structurally impossible.

**Weaknesses (ordered by production impact)**
1. **Incomplete fallback ladders (N-1).** The system degrades *downward* (paid → free) with sophistication, but has no *upward* rung (free exhausted → paid) even when budget and mandate permit. Degradation without escalation is a one-way ratchet to zero output.
2. **Single-process execution (N/A grade 17).** All parallelism is logical, none is physical. Fairness ≠ concurrency.
3. **Self-observation overhead (N-8).** 97% of queue capacity spent on heartbeats, ticks, syncs and killswitch checks.
4. **Silent failure paths.** Paused prep skips on provider failure by design ($0 > progress) but records nothing (N-7). Health writes fail without complaint (N-3). Two `except: pass` classes that hide the truth.
5. **Duplicate spawn paths (N-5).** The re-plan route lacks the idempotency discipline applied everywhere else.
6. **No end-to-end integration test.** Every unit passes; the product has never completed one full cycle. The test suite proves the parts, never the whole.
7. **Deployment gap.** CI gates the code; nothing gates the *deployment*. No staging, no post-deploy smoke check, no automatic version assertion.

**Technical debt register (new, carried into the roadmap)**
- Legacy `pipeline/agent/` dead code still resolvable and imported in places (`from agent import llm`).
- `docs/AGENTS_ROSTER.md` still describes the retired 18-agent orchestrator.
- Self-scheduling chains can multiply across restarts (N-6).
- SLA deadline default does not match the stated business rule (N-10).

**Engineering grade: 7.2/10** — strong fundamentals, disciplined process, one systemic blind spot
(escalation) and one architectural ceiling (single worker) between it and production readiness.
